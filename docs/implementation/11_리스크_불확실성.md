## 2. 마일스톤별 리스크 요약

### M1 — 스캐폴딩
- pnpm + Prisma 심링크 호환 (low)
- WSL2 Docker socket 마운트 (mid)
- Next.js 14 App Router 학습 곡선 (mid)

### M2 — 인증·조직
- 조직 격리 누락으로 정보 유출 (**high**) — scopedPrisma 헬퍼 + 코드 리뷰
- JWT 시크릿 운영 노출 (high) — Vault/KMS Phase 4
- 로그인 brute-force (mid) — rate limit Phase 4

### M3 — 프로젝트 관리
- 사설 repo 인증 (mid) — Phase 2 deploy key UI
- 테스트 파일 파싱 엣지 케이스 (low) — AST Phase 4
- 큰 repo clone 타임아웃 (mid) — `--depth=1` 기본

### M4 — 테스트 실행 엔진 (**최고 위험**)
- Docker 소켓 루트 권한 (**R2**)
- 진행도 파싱 정확성 (**R3**)
- BullMQ duplicate job (**R5**)
- Queue Router 추상화 미흡 → Phase 3 리팩터링 (mid)
- SSE 재연결 이벤트 유실 (mid) — Phase 2 Redis Stream + `Last-Event-ID` 헤더 기반 리플레이

### M5 — 리포트 조회
- 리포터 포맷 변경 (**R3**)
- 대용량 리포트 메모리 (**R9**)
- Trace 파일 public 노출 (mid) — 서명 URL Phase 2

### M6 — Claude Agent SDK (**최고 위험**)
- SDK API 변동 (**R1**)
- AsyncIterator 동시성 race (mid) — 스파이크 + 단위 테스트
- cwd 이탈 (Bash 도구) (high) — Phase 1은 Bash 제외
- 비용 폭주 (**R6**)
- working/stable 일관성 (**R7**)
- sessionId 관리 (mid) — init 메시지 수신 시 write-back
- 프론트 상태 머신 (**R10**)

### M7 — 대시보드·폴리싱
- 대시보드 쿼리 성능 (mid) — Redis 캐시 + 인덱스
- 초기 데이터 부족 차트 (low) — empty state
- E2E flaky (mid) — retries 2 + 명시적 wait
- 배포 재현 실패 (mid) — clean VM 주기 검증

## 3. 기술 스택별 불확실성

### 3.1 Claude Agent SDK (**최상**)

- **불확실한 항목**:
  - `AsyncGenerator`의 `interrupt()` 호출 후 cleanup 순서
  - `rewindFiles` 실제 동작 (파일 checkpoint 저장 위치, dryRun 결과)
  - `listSessions` / `getSessionMessages`의 세션 저장 위치 (메모리 vs 디스크)
  - 재시작 시 세션 재개 가능성
  - 스트리밍 이벤트 타입(`stream_event`의 정확한 sub-types)
  - `persistSession: true` 옵션의 지속성 메커니즘
- **선행 조치 (M6 착수 1~2일 전)**: `scripts/sdk-spike/` 하위 6개 스크립트 작성·실행
  - `basic-query.ts` — 최소 호출, 응답 스트림 순회
  - `multi-turn.ts` — AsyncIterable input으로 후속 턴
  - `file-edit.ts` — cwd 고정, Read/Edit 도구로 실제 수정
  - `interrupt.ts` — interrupt 후 이벤트 종결 시점
  - `rewind.ts` — rewindFiles(dryRun: true, false) 결과
  - `sessions.ts` — listSessions/getSessionMessages 반환 구조
- **결과 문서화**: 본 리스크 문서에 "SDK 스파이크 결과" 섹션 추가 (스파이크 수행 후)

### 3.2 BullMQ + Redis (중)

- **불확실한 항목**:
  - Redis 재시작 시 in-flight job 상태 (`lockRenewTime`, stalled detection)
  - Phase 3 다중 Redis 동기화 방식
  - queue events 전파 시점
- **선행 조치**: M4 착수 시 BullMQ 5.x 공식 문서 재확인, 스폿 체크

### 3.3 Dockerode (중)

- **불확실한 항목**:
  - 컨테이너 로그 stream demux 안정성 (8바이트 헤더 파싱)
  - Windows/WSL2에서 `--network=none`, `/var/run/docker.sock` 동작
  - `container.wait()` vs `attach()` 스트림 타이밍
- **선행 조치**: M4-S2에서 alpine 이미지로 단위 테스트 작성 (짧은 echo, abortSignal)

### 3.4 Next.js 14 App Router + SSE (EventSource) (중)

- **불확실한 항목**:
  - Server/Client Component 경계에서 EventSource 연결 패턴 ('use client' 필수)
  - Streaming SSR과 TanStack Query 조합
  - HMR 시 EventSource 중복 연결 방지 (useEffect cleanup)
  - EventSource의 커스텀 헤더 제약 — JWT는 쿼리 파라미터(`?token=`) 또는 쿠키로 전달
- **선행 조치**: 첫 커스텀 훅 `useRunEvents` 구현 시 패턴 확립

### 3.5 Prisma 5 + PostgreSQL 16 (저)

- 비교적 안정적. pnpm 심링크 이슈만 주의.

## 5. Claude SDK 스파이크 체크리스트 (M6 착수 전)

### 5.1 환경 준비
- [ ] `@anthropic-ai/claude-agent-sdk` 최신 설치
- [ ] `CLAUDE_API_KEY` 유효성 확인 (간단 호출 성공)
- [ ] 실험용 `projects/working/test-project/` 생성 (샘플 Playwright 스펙 1개)

### 5.2 검증 항목
- [ ] `query({ prompt: "Hello" })` 최소 호출 — 응답 AsyncIterator 순회
- [ ] 멀티턴: `prompt: AsyncIterable<SDKUserMessage>` 형태로 2회 이상 입력
- [ ] 도구 사용: `allowedTools: ['Read','Edit']`로 파일 수정
- [ ] `includePartialMessages: true` 효과 확인 (stream_event 발생)
- [ ] `enableFileCheckpointing: true`와 `rewindFiles(messageId, { dryRun: true })` 결과
- [ ] `abortController.abort()` 또는 `queryInstance.interrupt()` 호출 후 정리 시점
- [ ] `resume: sessionId` 옵션으로 이전 세션 이어가기
- [ ] `listSessions(cwd)` 반환 구조 (sessionId 목록)
- [ ] `getSessionMessages(sessionId, cwd)` 반환 구조
- [ ] `result` 이벤트의 `subtype`, `usage`, `total_cost_usd` 필드

### 5.3 실패 시 결정
- 핵심 기능 미지원 발견 시 → M6 스코프 축소(예: rewindFiles는 Phase 2로, interrupt만 Phase 1)
- SDK 설치/인증 실패 → Claude API 직접 사용으로 fallback (스펙 08 재작성 필요)
